# 소셜 프로필 > 메시지 수발신 기능 도입 검토

작성일: 2026-04-25  
대상: 소셜 프로필(친구/팔로우 기반) 내 1:1 메시지 기능(텍스트 중심)

## 1) 검토 목적
- 소셜 프로필 화면에서 사용자 간 직접 대화(메시지 수발신) 기능을 제공할지 판단한다.
- 도입 시 필요한 **시스템 설계**, **Google Play 스토어 등록/정책 대응**, **법적 리스크**를 사전에 정리한다.
- MVP(최소기능) 범위를 명확히 하여 출시 리스크와 운영 비용을 통제한다.

## 2) 기능 범위 제안 (MVP)
### 포함
- 1:1 텍스트 메시지 송수신
- 친구/팔로우 관계 기반 대화 시작(임의 사용자 DM 제한)
- 신고/차단/대화 숨김
- 욕설·유해 키워드 1차 필터(경고 또는 전송 차단)
- 최근 대화 목록 및 읽음 여부(읽음 시간은 선택)

### 제외(후속)
- 이미지/파일 전송
- 음성/영상 통화
- 오픈 채팅방/그룹 채팅
- 종단간 암호화(E2EE)

## 3) 기대 효과 vs 주요 우려
### 기대 효과
- 앱 체류시간 증가(대화 리텐션)
- 소셜 관계 강화로 재방문 빈도 상승
- 챌린지/퀘스트 협업 유도 가능

### 주요 우려
- 실시간 인프라 비용 증가(Firestore 읽기/쓰기, 푸시 발송)
- 스팸/괴롭힘/불법 콘텐츠 대응 부담 급증
- 미성년자 보호 및 개인정보 처리 이슈 확대

## 4) 시스템 관점 검토
## 4-1. 아키텍처 초안
- 저장소: Firestore
  - `conversations/{conversationId}`: 참여자, 최신 메시지, 상태값
  - `conversations/{conversationId}/messages/{messageId}`: 본문, 발신자, 생성시각, 상태
  - `userConversations/{uid}/items/{conversationId}`: 사용자별 인덱스(목록 조회 최적화)
- 서버: Cloud Functions
  - 메시지 쓰기 검증(권한, 차단 여부, rate limit)
  - 콘텐츠 필터링(금칙어/URL/반복문구)
  - 신고 접수 및 운영자 큐 적재
- 푸시: FCM
  - 앱 포그라운드/백그라운드 분기
  - 과도 알림 방지(쿨다운, 묶음 전송)

## 4-2. 보안/권한 원칙
- Firestore Rules로 참여자 외 읽기/쓰기 차단
- 차단 관계(`blocks`)가 존재하면 전송 차단
- 1인당 전송 속도 제한(예: 분당 N건) 및 비정상 패턴 탐지
- 메시지 삭제 정책(사용자 삭제 vs 서버 보존 로그) 분리

## 4-3. 운영 관점 필수 기능
- 신고 카테고리(욕설/성희롱/사기/개인정보요청 등)
- 차단 즉시 효력 및 기존 대화 접근 제한
- 관리자 콘솔: 신고 triage, 제재(경고/일시정지/영구정지)
- 감사 로그: 제재 근거 보관(분쟁 대응)

## 4-4. 성능/비용 고려
- 대화 목록은 denormalization(사용자별 인덱스)로 읽기 비용 절감
- 메시지 pagination(예: 20~50개 단위)
- TTL/아카이브 정책(장기 미사용 대화 압축 또는 삭제)
- 예상 급증 구간(이벤트/업데이트일) 기준 알림 트래픽 버퍼링

## 5) Play 스토어 등록/정책 대응 체크
> 아래는 출시 전 반드시 정책 원문과 Play Console 최신 문구로 재확인 필요.

### 5-1. 스토어 등록물(설명/스크린샷)
- 신규 핵심 기능 설명 반영
  - “사용자 간 메시지 기능”, “신고/차단 제공”, “안전한 커뮤니티 운영”
- 스크린샷 업데이트
  - 대화 목록, 대화창, 신고/차단 화면
- 데이터 안전(Data safety) 섹션 점검
  - 수집/처리 데이터 유형: 사용자 콘텐츠(메시지), 식별자, 신고 정보

### 5-2. 정책 이슈 포인트
- UGC(사용자 생성 콘텐츠) 앱 요건 충족 필요
  - 신고 기능, 차단 기능, 제재 프로세스, 운영 정책 고지
- 아동/청소년 대상 가능성이 있으면 추가 보호장치 필요
  - 성적/폭력/괴롭힘 관련 콘텐츠 대응 강화
- 개인정보처리방침/이용약관 업데이트
  - 메시지 보관기간, 신고 데이터 처리, 제3자 제공 여부

### 5-3. 출시 게이트(내부)
- [ ] Play Console 앱 설명/스크린샷 반영
- [ ] Data safety 문항 재작성 및 법무 검토
- [ ] 신고·차단 동작 QA(회피 불가 여부) 완료
- [ ] 정책 위반 콘텐츠 대응 SLA 수립(예: 24~72시간 내 1차 처리)

## 6) 법적 리스크 검토(한국 서비스 기준 초안)
> 법률 자문 대체가 아니며, 출시 전 전문 변호사 검토 권장.

### 6-1. 개인정보보호 리스크
- 메시지 본문은 개인정보 포함 가능성이 높아 민감한 분쟁 발생 여지 큼
- 최소수집 원칙 위반 우려: 불필요한 메타데이터 장기 보관 금지
- 대응
  - 보관기간/파기정책 명시(예: 탈퇴 후 N일, 신고건 예외)
  - 접근통제 및 운영자 열람 권한 최소화
  - 전송/저장 구간 암호화 및 로그 마스킹

### 6-2. 명예훼손/모욕/스토킹·협박 등 불법행위 리스크
- 플랫폼이 분쟁 채널로 사용될 수 있으며 신고 대응 지연 시 평판/법적 부담 증가
- 대응
  - 신고 접수 즉시 임시조치(노출 제한) 트리거
  - 반복 가해자 제재 정책(누적 점수 기반)
  - 수사기관 요청 대응 프로세스(증적 보전/제출 절차) 문서화

### 6-3. 청소년 보호 리스크
- 미성년자 간/성인-미성년자 간 접촉 이슈 발생 가능
- 대응
  - 연령대 기반 DM 제한(옵트인/보호자 동의 모델 검토)
  - 의심 키워드 탐지 및 안전 알림
  - 야간 알림 제한/상대 비공개 옵션 제공

### 6-4. 약관/고지 미비 리스크
- 메시지 운영정책 미고지 시 제재 정당성 약화
- 대응
  - 이용약관에 금지행위·제재·이의신청 절차 명확화
  - 개인정보처리방침에 메시지 처리 목적/보관기간/파기 절차 반영

## 7) 권장 출시 전략
### 7-1. 단계적 롤아웃
1. 내부/클로즈드 테스트: 직원·테스터 한정
2. 소규모 퍼센트 롤아웃(예: 5% → 20% → 50% → 100%)
3. 단계별 지표 충족 시 확대

### 7-2. 핵심 성공/중단 지표
- 성공 지표
  - 메시지 전송 성공률
  - DAU 대비 대화 참여율
  - 신고 처리 리드타임
- 중단(rollback) 트리거
  - 스팸/악성신고 급증
  - 크래시율/지연 급증
  - 정책 경고 또는 스토어 심사 리젝

## 8) 의사결정 제안
- **결론(권고): 조건부 도입**
  - 조건 1: 신고·차단·제재 운영체계 선구축
  - 조건 2: 약관/개인정보처리방침 및 Data safety 업데이트 선완료
  - 조건 3: 이미지/파일 전송은 2차로 유예(법적·운영 리스크 축소)
