### 1-2) 현재 구조에서 이미 분리된 축

현재 `app.js`에는 아래와 같은 분리 힌트가 이미 반영되어 있음.

1. 상단 정적 import: `network-monitor`, `conversion-tracker`, `performance-monitor`.
2. 하단 동적 import: `ad-manager`, `social`, `exercise-calc`, `pomodoro`, `meditation`, `library`, `movie`, `notification`, `dday`, `life-status`, `challenge-roulette`, `reels`, `rating-manager`.
3. 본문 내 “`modules/...로 분리됨`” 주석 구간 존재(소셜/명언/릴스/D-Day/Life Status/챌린지).
4. `Module Bridge` 구간을 통해 외부 모듈 접근 지점이 이미 형성됨.

즉, 완전 신규 리팩토링보다 **브리지 중심 점진 분리**가 가장 안전함.

## 2. 분리 전략 원칙 (영향도 최소화)

1. **HTML 이벤트 계약 유지**
   - `onclick="..."`와 연결된 전역 함수 이름은 즉시 변경하지 않음.
   - 도메인 모듈 내 구현을 옮기되, 기존 `window.xxx`는 얇은 wrapper(호출 위임)로 유지.

2. **초기화 순서 고정**
   - Firebase/AppCheck/RemoteConfig/Auth 흐름의 상대 순서를 변경하지 않음.
   - 초기화 로직 추출 시 `initXxx()` 인터페이스로 감싸고, 엔트리에서 기존 순서대로 호출.

3. **브리지 우선, 상태 직참조 후순위 제거**
   - 단기: `window.AppModuleBridge`를 통해 필요한 함수/상태만 노출.
   - 중기: 브리지 API를 명시 타입(혹은 계약 문서)으로 고정 후 직접 전역 참조 축소.

4. **도메인 단위 배포 가능성 확보**
   - 한 번에 큰 이동 금지. 1~2개 도메인씩 추출 후 배포/관찰.
   - 각 단계마다 “기능 검증 체크리스트 + 롤백 포인트”를 정의.

## Phase 0. 안전장치 선행 (1일)

**목표**: 리팩토링 전에 회귀 감지 체계를 먼저 확보.

### 작업

- 콘솔 에러 수집 규칙 고정(초기 로드/탭 전환/로그인 직후 3시점).
- `app.js`의 `window.*` export 목록 스냅샷 문서화.

### 완료 기준

### 롤백

## Phase 1. Core Shell 분리 (2~3일, 낮은 위험)

**대상**: 도메인 로직이 아닌 기반 로직.

### 작업
1. `modules/core/bootstrap.js` 생성
   - Firebase/AppCheck/Analytics/RemoteConfig/Network 연결을 함수 단위로 래핑.
2. `modules/core/app-state.js` 생성
   - `getInitialAppState`, 상태 초기화/읽기 보조 함수 이동.
3. `modules/core/nav-ui.js` 생성
   - 탭 순서 저장/로드, 네비 재정렬 로직 이동.

### 적용 방식
- `app.js`에서는 기존 함수명을 유지하고 내부 구현만 import 호출로 위임.
- 외부 동작/전역 심볼명은 유지.

### 완료 기준
- 앱 최초 로드, 로그인, 탭 이동, 상태창 카드 편집이 기존과 동일하게 동작.

### 롤백 포인트
- `app.js`에서 새 core import 제거 후 기존 인라인 구현 복원(커밋 단위 롤백 가능).

## Phase 2. 저결합 도메인 선분리 (3~4일, 낮은~중간 위험)

**대상 우선순위**: 이미 분리 신호가 있는 영역 + 의존도가 낮은 영역.

1. `onboarding` (가이드 UI)
2. `streak/rare-title` (상태 갱신 중심)
3. `quest-history/statistics` (렌더링 + 집계)

### 작업 방식
- 각 도메인마다 `init()`, `bindWindowHandlers()`, `render()` 3계층 구조로 정리.
- `window` 연결은 `bindWindowHandlers()`에서만 수행.

### 완료 기준
- 온보딩 노출, 스트릭 증가/표시, 퀘스트 히스토리 렌더가 기존과 동일.

### 롤백 포인트
- 도메인별 파일 단위 revert 가능(상호 결합 최소화).

## Phase 3. 고위험 도메인 분리 (46일, 중간~높은 위험)

**대상**
- 인증/프로필(`auth/profile`)
- 광고(`ad-manager` 연동부)
- 플래너(캘린더/사진)
- 권한/디바이스 연동(GPS/건강/푸시)

### 리스크 제어
1. 인증: 사용자 세션 경계(onAuthStateChanged)만 먼저 분리 후 세부 기능 이동.
2. 광고: 보상 지급 함수는 “단일 승인 함수”로 집중하여 중복 지급 방지.
3. 플래너: 날짜 계산 유틸을 먼저 공통화 후 UI 이동.
4. 권한/푸시: 플랫폼 분기(web/native) 계약을 별도 모듈로 고정.

### Phase 3-A. 권한/디바이스 연동(GPS/건강/푸시) 구현 상세 (권장 15일)

> 참조 기준:  
> - `docs/app-js-phase0-baseline-2026-04-22.md`  
> - `docs/app-js-phase0-window-export-스냅샷-2026-04-22.md`  
> - `docs/app-js-phase0-도메인별-수동-QA-체크리스트-2026-04-22.md`

#### 목표
- 권한 요청/상태 조회/설정 이동 흐름을 단일 계약으로 통합한다.
- `window` 직접 접근을 줄이고, 브리지 함수로만 호출되도록 정리한다.
- 권한 거부/부분 허용/네트워크 실패에서도 앱이 루프 없이 복구되게 한다.

#### 모듈 설계
1. `modules/device/platform-capabilities.js`
   - `isNativePlatform`, `isWebPlatform`, `supportsHealth`, `supportsPush`, `supportsGps`.
2. `modules/device/permission-service.js`
   - `getPermissionState(type)`, `requestPermission(type)`, `openAppSettings()`.
   - `type`: `gps | health | push`
3. `modules/device/push-service.js`
   - 토글 저장/복원, 중복 요청 방지(`_pendingPermissionPrompts` 대체), 인앱 알림 연결.
4. `modules/device/health-service.js`
   - 건강 데이터 접근 가드, 미지원 플랫폼 fallback 메시지 처리.
5. `modules/device/location-service.js`
   - GPS 읽기/실패 코드 정규화(`denied`, `timeout`, `unavailable`) 및 재시도 정책.

#### 구현 순서 (일 단위)
1. **D1~D3: 계약 고정**
   - `permission-service` 인터페이스 우선 정의.
   - 기존 `window.openAppSettings`, `window.updateCameraToggleUI`, 푸시 토글 호출 경로를 브리지 래퍼로 전환.
2. **D4~D7: Push 분리**
   - 권한 요청 중복 방지 큐를 모듈 내부 상태로 이동.
   - 푸시 ON/OFF 저장/복원 로직을 모듈화하고 기존 UI 이벤트는 wrapper 유지.
3. **D8~D11: GPS 분리**
   - 위치 요청, 실패 코드 처리, 사용자 안내 문구를 `location-service`로 일원화.
   - web/native 분기를 `platform-capabilities` 기반으로 단순화.
4. **D12~D15: 건강 연동 분리**
   - 지원 플랫폼에서만 건강 동기화 버튼 활성화.
   - 미지원/권한 거부 시 기능 강등(degrade) 동작 명시.

#### 운영 안전장치
- Baseline 문서의 3시점 수집 규칙(초기 로드/탭 전환/로그인 직후)을 동일 적용.
- `window.*` export는 즉시 제거하지 않고, Phase 3 동안은 wrapper만 slim화.
- 권한 요청 함수는 전부 idempotent 하게 구성(동일 세션 중복 호출 시 no-op).
- 에러 로깅 키를 고정:
  - `permission.request.failed`
  - `push.toggle.failed`
  - `gps.read.failed`
  - `health.sync.failed`

#### 검증 체크리스트 (Phase 0 QA 확장)
- 푸시 권한 팝업이 중복 호출되지 않는다.
- 푸시 토글 ON/OFF가 재진입 후에도 유지된다.
- GPS 권한 거부 상태에서도 탭 이동/기타 기능이 정상 동작한다.
- 건강 연동 미지원 기기에서 오류 대신 안내 UI가 노출된다.
- 로그인 직후 30초 내 권한/Auth 관련 신규 콘솔 에러가 없다.

#### 롤백 단위
- `push-service`, `location-service`, `health-service`를 각각 독립 커밋으로 분리.
- 장애 발생 시 도메인 단위 revert 후 wrapper 경유 기존 경로로 즉시 복귀.

### 완료 기준
- 로그인/로그아웃, 보상형 광고 보상, 플래너 저장/불러오기, 푸시 토글이 회귀 없음.

### 롤백 포인트
- 인증/광고/플래너/권한 모듈을 각자 독립 커밋으로 분리해 부분 롤백 가능하게 유지.

## Phase 4. app.js 엔트리 최소화 (1~2일, 낮은 위험)

**목표**: `app.js`를 “오케스트레이터(초기화 + 모듈 wiring)”로 축소.

### 작업
- `app.js`에서 도메인 구현 제거, 아래 항목만 유지
  - 환경 초기화
  - 모듈 import
  - DOMContentLoaded에서 init 순차 호출
  - 예외 로깅/장애 가드

### 완료 기준
- `app.js`의 역할이 명확히 엔트리로 한정되고, 신규 기능 개발이 도메인 모듈에서만 진행.

## 6. 결론

현재 코드는 이미 모듈 분리의 기반(동적 import + module bridge)을 갖추고 있으므로,
**“브리지 유지 → 도메인별 점진 추출 → 엔트리 최소화”** 순서가 가장 안전하다.

특히 영향도 최소화를 위해서는,
- 기능별 대규모 일괄 이동보다
- 도메인 1~2개씩 분리하고
- 단계별 검증/롤백 포인트를 고정하는 방식이 적합하다.
