## 2. 이용자 시나리오 정의

### 시나리오 A: 라이트 유저 (하루 1회 접속, 퀘스트만)
- 앱 실행 → 로그인 상태 확인 → 유저 문서 읽기 → 퀘스트 완료 → 저장
- 푸시 알림 수신 (레이드 3회 + 리마인더 1회)

### 시나리오 B: 일반 유저 (하루 2-3회 접속, 플래너+릴스 활용)
- 시나리오 A + 플래너 작성/수정 + 릴스 1개 포스팅 + 다른 유저 릴스 조회/좋아요
- 사진 업로드 1-2장 (플래너 or 릴스)

### 시나리오 C: 헤비 유저 (하루 5회+ 접속, 소셜 기능 적극 활용)
- 시나리오 B + 친구 프로필 조회 + 릴스 댓글/좋아요 다수 + 프로필 사진 변경

## 6. 비용 급증 위험 시나리오

### 6.1 릴스 피드 과다 조회
- ~~릴스 사진이 CDN 캐시 없이 직접 Storage에서 로드될 경우~~
- ~~DAU 15,000 × 평균 20장 조회 × 1.5MB = **450GB/일 다운로드**~~
- ~~**월 $1,600+ 가능**~~
- ✅ **완화됨**: CDN 캐싱(Storage 공개 읽기 + Cache-Control) 및 Service Worker 이미지 캐싱 적용으로 동일 이미지 반복 다운로드 제거. CDN 히트 시 Storage 원본 요청 발생하지 않음.

### 6.2 getDocs() 전체 컬렉션 스캔
- `app.js`에서 `getDocs(collection(db, "users"))` 호출 시 모든 유저 문서를 읽음
- 유저 10만 명이면 1회 호출 = 100K 읽기
- 관리자가 아닌 일반 유저에서 호출되면 비용 폭증 위험

### 6.3 스트릭 경고 푸시 (sendStreakWarnings)
- 비활성 유저가 많을 경우 매일 전체 유저 스캔 + 개별 FCM 전송
- 유저 10만 명 → 10만 읽기 + 조건부 쓰기 매일

### 6.4 FCM 토큰 정리 (cleanupInactiveTokens)
- 매주 전체 유저 스캔 → 유저 수만큼 Firestore 읽기

## 11. CDN 캐싱 적용 현황

> 적용일: 2026-03-24 (PR #253)

### 적용된 변경 사항

| 파일 | 변경 내용 |
|------|----------|
| `firebase.json` | Hosting Cache-Control 헤더 추가 (JS/CSS 30일, HTML/SW no-cache, 이미지 30일, 폰트 1년 immutable) |
| `storage.rules` | `reels_photos`, `profile_images` 공개 읽기 전환 → Firebase Storage CDN 캐싱 활성화 |
| `app.js` | 업로드 시 `cacheControl` 메타데이터 설정 (릴스 24h, 프로필 7일, 플래너 비공개) |
| `sw.js` | `googleapis.com` 포괄 우회 제거 → Storage 이미지 전용 Cache First 전략 (24h 만료, 150개 제한) |

### 캐싱 계층 구조

```
요청 흐름:
  브라우저 → [1] Service Worker Cache (24h, 150개)
           → [2] Firebase Storage CDN 엣지 (Cache-Control 기반)
           → [3] Cloud Storage 원본 (asia-northeast3)

캐시 히트 시 [3] 원본 요청 없음 → Storage 다운로드 비용 $0
```

### 보안 고려사항

- `reels_photos/`: 공개 읽기 허용 — 소셜 피드에 이미 노출, 24시간 후 자동 삭제, URL에 추측 불가능한 토큰
- `profile_images/`: 공개 읽기 허용 — 피드에서 모든 유저에게 노출되는 반공개 콘텐츠
- `planner_photos/`: **인증 필수 유지** — 개인 콘텐츠

### ✅ 적용 완료 (Phase 2)

- `generateThumbnail` Cloud Function (`sharp` 기반 240px WebP 리사이즈, `onObjectFinalized` 트리거)
- 썸네일 저장 경로: `thumbs/{원본경로}` (예: `thumbs/reels_photos/uid/123.webp`)
- 피드에서 썸네일 URL 사용 (이미지당 ~1.5MB → ~40KB), 로드 실패 시 원본 fallback
- 기존 cleanup 함수들에 썸네일 동시 삭제 로직 추가
